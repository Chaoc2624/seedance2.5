# Accounts & Isolation Reference

外链提交体系里最容易被忽视、却决定成败的一层：**身份策略**（用什么邮箱 / 哪些 OAuth 号）和**隔离策略**（每个站一个独立的浏览器环境）。

本文件只给**原则和取舍**。具体凭据（谁的邮箱、哪个 Gmail、代理端口号）由调用方项目管理——这是铁律 scope 外，参见 `SKILL.md` "Out of Scope"。

## 邮箱策略

### 四类邮箱与适用场景

| 类型 | 适用场景 | 风险 |
| --- | --- | --- |
| 品牌域名邮箱（`team@yourbrand.com`） | SaaS 目录要求域名归属验证、Guest Post 联络站长 | Akismet / CleanTalk 信誉惩罚，日常评论提交必输 |
| 品牌域名 catch-all（`anything@yourbrand.com`） | 理论上一站一地址方便管理 | 大量反垃圾系统对 catch-all 邮箱打低分 |
| 干净 Gmail（全新注册、正常使用痕迹） | 95% 的外链提交场景 | 无（信誉干净） |
| Gmail plus-addressing（`you+sitename@gmail.com`） | 需要一站一地址（追踪、多账号隔离） | 某些严格反垃圾会识别 `+` 为 spam 模式 |

### 决策流程

1. 需要目标站验证"邮箱属于品牌域名"（归属验证型提交） → 品牌邮箱
2. 目标站是 WP 评论 / 普通注册 / 论坛 Profile → **干净 Gmail**（默认选项）
3. 干净 Gmail 已经在该站用过，需要第二个地址 → **Gmail plus-addressing**
4. 品牌 catch-all 在试过之后静默失败 → 立刻切 Gmail（铁律 9）

### 干净 Gmail 的"干净"标准

**不干净 Gmail 的特征**：

- 该 Gmail 曾在很多 spam / 推销场景使用（即使是个人使用，广告投递也会让 Google 给该地址打分）
- 该 Gmail 曾注册过大量 forum / 评论站（外部反垃圾库会学习这个信誉）
- 该 Gmail 的 recovery 邮箱是另一个黑名单邮箱

**干净 Gmail 的准备**：

- 新注册的 Gmail，建立日期超过 30 天（规避"新号风险"评分）
- 有少量真实使用痕迹（Gmail 收发几封正常邮件、订阅几个正经 newsletter）
- 不用 VPN / 不常用代理登录（IP 多样化反而是 spam 信号）
- 账号资料（头像、展示名）填一致，不要留空

## OAuth 账号管理

一些平台只接 Google / GitHub / Twitter 登录（velog、dev.to 等）。OAuth 账号处理：

### 原则

- **OAuth 账号的信誉外溢**：同一个 GitHub 号登录 dev.to 发 spam，你的 GitHub commit 历史会出现在 dev.to Profile 上。反过来，你的 GitHub 账号会被标记为可疑。要避免这种溢出。
- **专用 OAuth 账号池**：如果做大批量，为外链工作准备专用 GitHub / Google 账号，与日常开发账号隔离。
- **OAuth 账号的真实性**：新开的 GitHub 号没有 contribution graph，容易被平台识别为"仅为登录而创建"。对策：让号有几个月的真实 star / fork 活动再用（不需要写代码，star 公开仓库即可）。

### 复用规则

- 同一品牌下的多个相关平台（velog + dev.to + Hashnode）可以复用同一 OAuth 账号，因为身份是一致的
- **跨品牌不要复用**——四个站的外链都用同一个 GitHub 号登录，平台一旦封号四个站全挂

## 多站并发隔离

当同时为多个品牌 / 多个主站做外链时，必须隔离。

### 需要隔离的维度

- **Cookie / Session**：每站独立的 Chrome Profile 目录。不共用。
- **IP**：每站独立代理端口 / 独立 ISP 代理身份
- **邮箱池**：每站独立的一批邮箱，互不复用
- **OAuth**：每站独立的 GitHub / Google 号（至少三个品牌三份）
- **浏览器进程**：每个 Playwright MCP / claude-in-chrome 实例属于一个品牌，不跨品牌

### 为什么不能混

- 站 A 的评论被 Akismet 标记 → 如果站 A 和站 B 共用 Gmail，站 B 下次评论也会被拦
- 站 A 的 IP 被 CleanTalk 拉黑 → 站 B 再访问任何 CleanTalk 保护站，也立刻 403
- 站 A 和站 B 的浏览器 Profile 共用 → cookie 交叉污染，某站登录状态会意外出现在另一站 → 锚文本识别错误（铁律 8 的根因）

### 隔离架构（概念层）

具体实现（settings.json / tmux 布局 / Chrome profile 路径 / 代理配置）由调用方项目管理；本 skill 不假设任何特定架构。但概念层必须落实以下几点：

1. 每个品牌有自己的 "workspace" 目录
2. workspace 内：浏览器数据目录 + 代理配置 + 账号登录状态存档 + 外链记录表
3. 同时工作时：每个 workspace 启动独立的 agent 进程，agent 之间互不感知
4. 切换品牌时：切换整个 workspace，不要在同一个进程里临时换代理/换账号

### 平行并发的好处

- 效率：同时跑 N 个品牌是单品牌的 N 倍速度（前提是足够的 Claude Code 账号容量 / 代理带宽）
- 隔离：一个品牌出问题不影响其他
- 试错：同一种实验可以在多个品牌分别跑，验证结论一致性

## 账号维护

账号不是发完就扔。长期维护能把平均外链质量再提 20-30%：

- **Profile 完善度**：注册后补充头像、个人简介、几条早期活动（关注/star/发一两条正经内容），再去发外链。纯"幽灵账号"容易被平台下次审核触发。
- **不要同时注册大量账号**：一天之内同一 IP 下注册 10 个 GitHub 账号 → 全部被 GitHub 风控标记。分散到几周。
- **账号记录**：记录每个账号的邮箱 / 注册日期 / 所属品牌 / 登录过的站 / 已知问题。不要靠记忆管理账号。

## 当 scope 外部资源缺失时

如果本 skill 的调用方没有提供：

- 可用的干净 Gmail → 询问用户或让用户先建号；**不要用可能是用户主邮箱的地址**
- 代理 → 可以不用代理跑（只适合低强度场景），但告知用户没有代理的风险
- OAuth 账号 → 同上

绝不编造邮箱 / 账号 / 代理 IP 去尝试。出问题时风险外溢到用户的真实账号。
