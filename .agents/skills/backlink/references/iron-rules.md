# Iron Rules Reference

每次执行外链任务**先读这份文件**。违反任何一条都会直接毁掉这条外链，或者污染后续一批链接的数据质量。每条规则的结构固定：**规则 / Why / 如何适用**。Why 部分记录的是已经发生过的事故或代价，不是假设。

## 1. 禁止设限（No shortcuts）

- **规则**：目标站要求写 800 字 guest post 就写；要求 20 字段注册就填；要求邮箱验证就去邮箱点；要求审核排期就等。唯一合法的跳过理由是：真正的付费墙、站已经下线、Cloudflare 硬封锁到访问不了首页。
- **Why**：最容易的外链往往是最没用的外链。能省的那几分钟成本，换来的是一条权重为 0 的链接，甚至被算法识别为垃圾站的邻居。本项目的数据：凡是"觉得太麻烦跳过"的站，后来手动补回去的 ROI 都高于当晚跑的"速通清单"。
- **如何适用**：提交前默认所有步骤都必须走完。想跳过时，先在 `dead-sites.md` 找证据（已归档死站）；找不到证据就老实做完。

## 2. 前端不行先逆向（Reverse-engineer before giving up）

- **规则**：按钮点了没反应、Modal 不弹出、表单 submit 后卡住不动——**第一反应不是跳过，而是打开 `reverse-eng.md` 按 5 步 SOP 找后端 API**。
- **Why**：现代站点大量用 Vue/React mount，前端失败率很高（mount 竞态、SSR hydration 错位、jQuery 和框架事件冲突）。直接调后端通常 3-5 分钟就能搞定，跳过反而损失一条可得的外链。
- **如何适用**：前端操作失败两次以后，停止重试前端，切到逆向 SOP。案例见 `reverse-eng.md`（SaaS 目录投票 / velog OTP）。

## 3. 候选筛选查 spam + traffic，别看 DR

- **规则**：从任何第三方指标源拿候选站时，最低过滤条件是 `is_spam=0` 且真实流量 ≥ 100。DR（Domain Rating）、DA、Ahrefs Rank 这类合成指标不作为准入标准。
- **Why**：DR 可以互刷。曾经一晚上跑了一批 DR50+ 候选，结果全部是 PBN 链接农场，实际流量为 0，所有链接权重归零，还污染了品牌的外链画像。traffic 是相对难造假的指标。
- **如何适用**：候选入库的那一层就硬过滤。不要在提交阶段靠眼睛判断——到那一步人已经疲劳，会放水。

## 4. 去重按域名，不按模板 ID

- **规则**：外链去重的唯一键是 eTLD+1（裸域名）。第三方数据源里同一个域名可能对应 5 个不同的模板 ID / URL 变体，全当成一个站。
- **Why**：同一个域名做第二条外链的边际收益极低，同时增加被审核员注意到刷链痕迹的概率。按模板 ID 去重是典型的"数据表结构驱动业务逻辑"错误。
- **如何适用**：候选筛选 SQL、工作日志、最终提交列表三个环节都按域名去重。

## 5. 查邮件必须开新标签页

- **规则**：绝对不允许在正在填表的标签页里 `navigate` 到邮箱站去拿验证码。新开标签页，不切走当前页面。
- **Why**：大量站点用 sessionStorage、meta refresh token、短 TTL 的 CSRF token 保护表单。离开再回来表单可能重置、token 过期、已填字段丢失。典型代价：填了 15 分钟的 guest post 表单，navigate 去查邮件，回来一片空白。
- **如何适用**：需要邮件验证码时，先保存当前页截图，再在**新标签页**打开邮箱。Playwright 用 `browser_tabs` 新建；claude-in-chrome 用 `tabs_create_mcp`。

## 6. rel 属性每次实测

- **规则**：提交完成后必须用 JS 验证 `rel` 属性。数据库里标的"这个站 Dofollow"只能当参考，不能当结论。
- **Why**：2026 年有一轮大规模降级（`dofollow-2026.md` 底部有清单）。很多曾经 Dofollow 的平台主动给 UGC 内容加了 `rel="ugc nofollow"`。如果不实测就写回，整个 Dofollow 占比指标都是假的。
- **如何适用**：使用 `SKILL.md` Core Workflow 第 6 步的 JS 片段。结果分三档：EMPTY = Dofollow；`ugc`（单独）= 看策略，多数算半权重；含 `nofollow` / `sponsored` = Nofollow。

## 7. 先读知识库再查 DB / 再去翻页面

- **规则**：接到一个目标平台，第一动作是在本 skill 的 references 里搜一次。平台 SOP、反垃圾特征、rel 修正、已知坑点、逆向成功的 API endpoint——都应该先从知识库读。
- **Why**：每次从零开始看页面是最昂贵的选项。知识库就是为了避免重复踩坑而存在的。另一面：读完知识库发现没有记录，说明这是新平台，现在就是补档的最好时机（对齐规则 9）。
- **如何适用**：接到平台名或 URL → grep references/ → 没命中再上手看页面 → 完工后把新发现追加回对应 reference 文件。

## 8. 切站必须确认产品

- **规则**：从一个目标站切到另一个目标站（例如从 A 项目外链切到 B 项目外链）之前，**显式重新读一次当前项目的品牌名、主域、锚文本白名单**。
- **Why**：曾经一晚上的 18 条外链，锚文本全部写错——A 站的锚文本写进了 B 站的目标。外链发出去之后锚文本几乎不可能改回来（要联系站长或重发评论，大多数站长不理）。这是本项目损失最惨重的单次事故。
- **如何适用**：每次切换 session / 切换目标项目时，先读项目品牌 fact sheet（由调用方提供），再开始填表。不要凭记忆。Playwright 可以把当前品牌名写进 page title overlay 作为视觉提醒。

## 9. catch-all 邮箱失败立刻切 Gmail plus-addressing

- **规则**：用品牌域名 catch-all 邮箱（如 `any@yourdomain.com`）注册时，如果站点"注册成功但收不到验证信" / "提交按钮静默无反应" / "验证信进垃圾箱且带 spam 警告"——立刻切 Gmail plus-addressing（`你的gmail+sitename@gmail.com`），不要重试三次。
- **Why**：大量反垃圾系统有"自定义域名邮箱 = 高风险"规则（尤其是中小站用的共享反垃圾库）。catch-all 邮箱在 Akismet、CleanTalk 的信誉库里评分低。重试只是浪费时间，切邮箱是 1 秒的事。
- **如何适用**：每个品牌预先准备好一个干净的 Gmail，plus-addressing 自动打标签不混乱。参考 `accounts.md` 的邮箱策略。

## 10. 验证码协作：必须先填完所有字段

- **规则**：遇到人工才能过的验证码时，**先填完表单里所有其他字段，滚动到验证码可见位置，再告诉人类"所有字段填完，只剩验证码"**。人类过完验证码立刻点提交，不等确认。
- **Why**：很多站的验证码是一次性 token（滑块验证、reCAPTCHA v2 的旧 response）。曾经填了一半字段就叫人类过验证码，验证码过了、提交时才发现漏了两个必填字段，表单重置，验证码作废，又要等 30 秒刷新。一次事故 = 浪费一次人工注意力。
- **如何适用**：Playwright 先 fill 所有可见字段 → `browser_evaluate` 滚动到验证码 → `browser_take_screenshot` → 通知人类 → 人类操作后 agent 立刻点提交按钮。详细协议见 `anti-spam.md` "人机协作"段。
